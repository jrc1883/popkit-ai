# ElShaddai Monorepo - Git Work Tree Strategy

**Purpose:** Isolated, parallel development across 10 apps with automatic cleanup and port management.

---

## Overview

Every `/popkit:dev work #N` automatically creates an isolated work tree, enabling:
- **Parallel development** - Multiple agents work on different work trees simultaneously
- **Safe async execution** - Power Mode agents don't interfere with each other
- **Automatic cleanup** - Stale work trees cleaned up on completion
- **Port isolation** - Dev servers on different ports, no conflicts
- **Monorepo awareness** - Each app has its own work tree namespace

## Naming Convention

Work tree names follow this pattern:

```
<issue-type>-<issue-number>-<slugified-title>
```

### Issue Types

| Type | Use Case | Example |
|------|----------|---------|
| `feature` | New capabilities | `feature-123-auth-system` |
| `bugfix` | Bug fixes | `bugfix-456-login-regression` |
| `refactor` | Code restructuring | `refactor-789-components` |
| `hotfix` | Urgent production fixes | `hotfix-999-security-patch` |
| `experimental` | Proof of concepts | `experimental-234-real-time` |
| `research` | Investigations/spikes | `research-567-performance` |
| `docs` | Documentation | `docs-890-api-guide` |

### Naming Rules

1. **Max length:** 50 characters
2. **No spaces:** Use hyphens
3. **Slugify title:** Remove punctuation, lowercase
4. **Keep issue number:** Enables traceability

**Examples:**
- Issue #123 "Add user authentication" → `feature-123-auth`
- Issue #456 "Fix login regression on mobile" → `bugfix-456-login-regression`
- Issue #789 "Refactor component library" → `refactor-789-components`

### Port Assignment for Parallel Work Trees

When creating work tree in `/popkit:dev work #N`:

```javascript
// For Genesis (base port 3002)
const basePort = 3002;
const workTreeIndex = 1; // First parallel work tree
const assignedPort = basePort + (100 * workTreeIndex);  // 3102

// Result: Work tree gets port 3102 for dev server
// Configure in .env.local: PORT=3102
```

**Pattern:**
- Main branch: Base port (3002)
- Parallel work tree 1: Base + 100 (3102)
- Parallel work tree 2: Base + 200 (3202)
- Parallel work tree 3: Base + 300 (3302)

## When Creating a Work Tree

### Auto-Generated by `/popkit:dev work #N`

When you run `/popkit:dev work #123`:

1. **Parse issue**
   ```bash
   gh issue view 123 --json number,title,body,labels,state,author
   ```

2. **Extract Work Tree Context**
   - App name (from issue body)
   - Work tree type (inferred from labels)
   - Issue title (for naming)

3. **Generate work tree name**
   - Example: `feature-123-auth-system`

4. **Create work tree**
   ```bash
   git worktree add .worktrees/feature-123-auth-system \
       -b feature/123-auth-system \
       main
   ```

5. **Set up environment**
   - Create `.env.local` with assigned port
   - Run `npm install` (if dependencies changed)
   - Run baseline tests

6. **Enter work tree context**
   - All subsequent work happens in this isolated branch
   - Dev server runs on assigned port
   - No interference with other parallel work

### Multi-App Work Trees

If issue affects multiple apps (e.g., shared package update):

```
elshaddai/.worktrees/
├── refactor-789-shared-types/
│   ├── packages/types/        # Symlink or direct edit
│   ├── apps/genesis/          # Symlink or direct edit
│   └── apps/optimus/          # Symlink or direct edit
```

**Strategy:** Create single work tree for shared code, reference from apps.

## Async Agent Work Tree Isolation (Power Mode)

When using Power Mode with parallel agents:

### Per-Agent Batches

```
Issue #123: "Add authentication system"
Power Mode: RECOMMENDED (3 agents)

Batch 1: Phase 1 - Discovery (all agents share main work tree)
  - code-explorer: codebase analysis
  - code-architect: architecture planning
  - researcher: auth library research

  Checkpoint: Review findings

Batch 2: Phase 2 - Implementation (agents get isolated work trees)
  - Agent 1: auth-service → .worktrees/feature-123-auth-service/
  - Agent 2: auth-ui → .worktrees/feature-123-auth-ui/
  - Agent 3: auth-tests → .worktrees/feature-123-auth-tests/

  Parallel execution with coordination

  Checkpoint: Code review, merge conflicts resolved

Batch 3: Phase 3 - Integration (back to main work tree)
  - Merge all work from isolated trees
  - Run full test suite
  - Final review
```

### Coordination Across Work Trees

Power Mode coordinator:
1. Tracks which agent owns which work tree
2. Prevents file conflicts via `.claude/popkit/work-tree-state.json`
3. Coordinates cleanup when batch completes
4. Handles merge conflicts automatically

## Cleanup Strategy

### Automatic Cleanup

After work completes (`/popkit:dev work #123` finishes):

1. **Verify all commits pushed**
   ```bash
   git log main..feature/123-auth-system  # Should be empty
   ```

2. **Delete work tree**
   ```bash
   git worktree remove .worktrees/feature-123-auth-system
   git branch -d feature/123-auth-system
   ```

3. **Update state tracking**
   - Remove entry from `.claude/popkit/work-tree-state.json`
   - Log cleanup timestamp

### Manual Cleanup

For stale work trees (abandoned for >7 days):

```bash
# List stale work trees
/popkit:worktree list --stale

# Remove specific work tree
/popkit:worktree remove feature-123-auth-system

# Prune all stale across monorepo
/popkit:worktree prune-all --stale
```

### Cleanup Verification

```bash
# Check for orphaned work trees (no matching git branches)
git worktree prune

# List remaining work trees
git worktree list

# Verify .worktrees directory is clean
ls -la .worktrees/
```

## Work Tree Safety Levels

Issues can specify async safety to guide Power Mode:

### Safety Levels

| Level | Meaning | Implication |
|-------|---------|-------------|
| **none** | No async safety concerns | Multiple agents can work in parallel freely |
| **requires-lock** | File-level locking needed | Agents must coordinate file access |
| **safe** | Async-safe by design | Agents work on completely separate files |

### Example in Issue Body

```markdown
## Work Tree Context

**App:** genesis
**Type:** Feature
**Safety Level:** safe  ← Multiple agents can work simultaneously
**Ports:** 3102, 3202   ← Two parallel dev servers possible
```

## Best Practices

### 1. **Always Use Issues First**
```bash
# Create issue with Work Tree Context
/popkit:issue create "Add real-time notifications"

# Then work on it
/popkit:dev work #45
```

### 2. **Keep Work Tree Context Updated**
If requirements change during development, update the issue:
```bash
/popkit:issue edit 45 --body "... Updated Work Tree Context ..."
```

### 3. **Monitor Port Usage**
```bash
# Before starting work
/popkit:worktree list --show-ports

# Or manually
netstat -ano | findstr :3102  # Check if port is in use
```

### 4. **For Monorepo Multi-App Work**
If work spans multiple apps:
```bash
# Create tracking issue for each app
/popkit:issue create "Update types in genesis" --label "part-of-#45"
/popkit:issue create "Update types in optimus" --label "part-of-#45"

# Link them together
/popkit:issue link 45 --epic
```

### 5. **Document Async Decisions**
In issue body, clearly state if work requires sequential execution:

```markdown
## Work Tree Context

**Safety Level:** requires-lock
**Reason:** Modifies shared database schema - only one migration can run at a time
```

## Troubleshooting

### Port Already in Use

```bash
# Find what's using the port
netstat -ano | findstr :3102

# Kill the process
taskkill /PID <PID> /F

# Or use assigned port directly in .env.local
PORT=3103  # Choose different port
```

### Work Tree Merge Conflicts

When merging work tree back to main:

```bash
# Check conflicts
git status

# Resolve in editor
# Then complete merge
git add .
git commit -m "Merge feature work tree"

# Or abort if needed
git merge --abort
```

### Stale Work Tree Won't Delete

```bash
# Remove forcefully
git worktree remove --force .worktrees/feature-123-stale

# Prune orphaned refs
git worktree prune
git gc
```

### Parallel Agents Interfering

Check work tree isolation in Power Mode:

```bash
# View work tree state
cat .claude/popkit/work-tree-state.json

# Verify file ownership
/popkit:worktree analyze --conflicts

# Separate agents to different work trees (manual fix)
/popkit:dev work #45 --phases discover
# Then in new terminal:
/popkit:dev work #45 --phases implement
```

## Integration with popkit:dev

### The `work` Subcommand Flow

```
/popkit:dev work #123

1. Fetch issue #123
2. Parse Work Tree Context
   ├─ app: genesis
   ├─ type: feature
   ├─ name: feature-123-auth
   └─ ports: [3102]
3. Create work tree: git worktree add .worktrees/feature-123-auth
4. Set up environment: create .env.local, npm install
5. Parse PopKit Guidance
6. Route through orchestrator (quick/full/power)
7. Execute workflow in work tree
8. On completion:
   ├─ Push commits
   ├─ Delete work tree
   └─ Present next actions
```

### The `power` Mode Flow

```
/popkit:dev work #45 -p

1. Same as above, plus:
2. Identify async safety level
3. Create work tree batching plan
4. For each batch of agents:
   ├─ Create isolated work tree per agent
   ├─ Run agents in parallel
   ├─ Monitor for conflicts
   └─ Merge back to main work tree
5. Final merge to main
6. Cleanup all work trees
```
